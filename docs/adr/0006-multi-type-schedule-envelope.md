# Multi-type Schedule envelope with per-type body tables

## Context

Nursery is the only printable Schedule today: a monthly grid of workers per service, auto-generated by a fairness scorer, rendered with a logo header, exported as PDF/JPG, and optionally sent as an image via Messages. The full feature lives under `/nursery/*` with its own four pages, its own settings table (`nursery_settings`), its own logo upload, and its own send-image dialog.

Two more Schedule types are arriving:

- **Special Music** — a multi-month grid of Sunday AM / Sunday PM singer assignments. No auto-generation (casting is a pastoral decision); same logo / send / export shape; configurable title prefix and footer text blocks at the bottom (scripture quote + reminders). Cell entries are conceptually identical to existing `special_music` rows with `status='will_perform'`.
- **Sunday School** (future, ~4-month scope) — same export/send/logo plumbing; bespoke body.

Three structural shapes were on the table:

- **Loose** — keep `nursery_schedules` as-is; build a parallel `special_music_schedules` set of tables; reuse only at the React component layer. Future Sunday School duplicates the pattern again. Server-side plumbing copy-pasted per type.
- **Fully generic single table** — one `schedules` + one `schedule_slots` table with flexible columns. Per-type generators write into the same shape.
- **Envelope + per-type body** — one `schedules` row owning what's common (title, scope, status, logo via settings, footer blocks via settings); each type keeps its own body table.

The logo lives today on `nursery_settings.logoPath`. The user's intent is to make it a global cross-schedule asset.

## Decision

A single **`schedules`** envelope table owns every printable Schedule across types. The envelope columns are: `id`, `schedule_type` (`nursery` | `special_music` | future `sunday_school`), `scope_kind` (`monthly` | `date_range`), `month` + `year` (monthly only), `scope_start` + `scope_end` (date-range only), `scope_label`, `status` (`draft` | `final`), `created_at`, `updated_at`.

`nursery_schedules` is **dropped** and its rows are migrated into `schedules` with `schedule_type='nursery'` and `scope_kind='monthly'`. `nursery_assignments.scheduleId` is repointed at `schedules.id` in the same migration.

**Special Music has no body table.** A Special Music schedule's cells are `special_music` rows queried by the schedule's `scope_start..scope_end` date range filtered to `service_type in ('sunday_am', 'sunday_pm')`. No FK from `special_music` to `schedules`. Multiple schedules can overlap a single special_music row. `special_music.song_title` becomes nullable so cells can be virtual-then-populated.

Title and footer blocks are **settings-only, no per-instance override**:

- `settings.schedulesLogoPath` — single global logo, all types
- `settings.schedules.<type>.titlePrefix` — per-type title prefix
- `settings.schedules.<type>.footerBlocks` — per-type JSON array of text blocks
- `settings.schedules.specialMusic.singerGroupIds` — multi-select of existing Groups; members union'd and deduplicated as the singer pool

The action verb across all Schedule types is **"New Schedule"** — not "Generate." Nursery's auto-fill happens as a side effect of creation; Special Music creates an empty envelope and materializes cells as the user edits.

The sidebar gets a new **"Schedules"** parent group containing: Nursery, Nursery Workers, Special Music, Settings. URLs at `/nursery/*` are preserved for stability; `/special-music/*` is new; settings moves to `/schedules/settings`.

Shared React primitives extracted from the existing nursery page:

- `<SchedulePreviewFrame>` — renders the logo header, title (`${titlePrefix} ${scopeLabel}`), status badge, **footer blocks read live from settings**, and slots a per-type body component in the middle. This is the JPG/PDF capture target.
- `<ScheduleActionsToolbar>` — Edit / Finalize / Reopen / Export PDF / Export JPG / Send buttons.
- `useScheduleExport` hook — image generation + PDF assembly + image inlining, currently lives in `nursery-schedule-view-page.tsx`.
- `<SendScheduleDialog>` — recipient picker + caption + send.
- `<ScheduleListCard>` — search + paginated table + "New Schedule" dialog wiring.

## Why

- **Hard to reverse:** the migration drops `nursery_schedules` and repoints its FK. Rolling back means a reverse migration and re-splitting any shared envelope-level code. A future contributor seeing the `schedules` table with mixed `monthly`/`date_range` rows might be tempted to "clean it up" by splitting per-type — this ADR is the breadcrumb that says don't.
- **Surprising without context:** `schedules` carries both `month` + `year` *and* `scope_start` + `scope_end`, with `scope_kind` selecting which pair is meaningful. A reader will (correctly) suspect that's nullable-soup; this file justifies the discriminator over forcing both kinds onto one shape. Also surprising: Special Music has *no body table* — the body is `special_music` rows queried by date range. The next contributor adding Sunday School will assume they need a body table; this file confirms that's right for Sunday School but explains why Special Music doesn't.
- **Real trade-off — we rejected two alternatives:**
  - **Loose (per-type parallel tables, share only React).** Lower migration risk, no schema upheaval. The cost is that every future Schedule type re-implements list/detail/export/send server plumbing. The two-types-in-the-codebase-now precedent would lock that pattern in. Cross-cutting changes (e.g., adding a "shared via link" feature to all schedule types) would touch N parallel implementations.
  - **Fully generic single-table.** Maximum reuse, but it forces Nursery's `(worker_id, slot, service_type, is_carryover)` and Special Music's `(performers[], guest_strings[], override_label, derived_type)` shapes into one row with most columns null per type. The generator function becomes a `switch (type)` from line one. The wins from "everything in one query" don't materialize because the per-type read paths look nothing alike anyway.

The envelope-plus-body model captures the genuine shared concept (the printable wrapper) without forcing the contents to match.

- **No FK from special_music → schedules** is itself a trade-off. It means a Special Music schedule is a *view* over a date range, not an *owner* of rows. Pro: deleting a schedule leaves the specials intact (you printed it, but the performers haven't disappeared); a special created from `/music/specials/new` auto-appears on any overlapping schedule; multiple schedules (e.g., a 2026 annual planner and a Summer 2026 wall poster) can coexist showing the same rows. Con: there's no "this row belongs to that schedule" affordance; row provenance is implicit in its `date` + `service_type`. We chose ownership-by-date-range because casting decisions naturally happen against a date range, not against a schedule-document identity.

## Consequences

- **`special_music.song_title` is nullable.** Existing rows are unaffected (all non-null). Any query/UI that previously assumed non-null must add a fallback (display "—" or hide the row from "song stats" rollups). The Specials list page should filter `will_perform` rows with null song from any "performed history" view.
- **Schedule status `draft`/`final` is the gate for export/send.** Reused from nursery. Special Music adopts the same lifecycle: Draft = editable + send/export disabled; Final = locked + send/export enabled; "Reopen as Draft" toggles back. No per-type variation.
- **Settings is the source of truth for footer/title.** Re-exporting an old finalized schedule months later renders today's footer text (since there's no per-instance copy). This is acceptable because the footer is house style ("our church always prints this Psalm"), not historical content. Documented here so a future "but my old PDF said something different" report routes to "yes, settings changed; the original PDF you mailed is unaffected."
- **The logo file moves.**  `$UPLOADS_DIR/nursery-logos/` is renamed to `$UPLOADS_DIR/schedule-logos/` in the migration. `nursery_settings.logoPath` is migrated to `settings.schedulesLogoPath`. `nursery_settings` table is otherwise unused after consolidation and can be dropped; verify no other code references it before removal.
- **`<NurserySchedulePreview>` loses its title header.** Title + logo move into `<SchedulePreviewFrame>`. The preview component renders only the table body inside the frame's body slot. Visual fidelity must be verified: the existing nursery PDF/JPG should look pixel-identical after the refactor.
- **No carryover concept for Special Music.** Nursery's 5-Sundays-borrow-from-prior-month logic stays Nursery-specific in `nursery-scheduler.ts`. Special Music's date range is taken literally — `scope_start` and `scope_end` define exactly which Sundays render. Future schedule types decide their own scope semantics.
- **`schedules.schedule_type` is closed.** New schedule types require a code change (migration to extend the enum, new route module, new body table or query strategy, new preview component). This is acceptable; "schedule types" is a small, slow-growing set.
- **The "New Schedule" verb is shared.** Every list page uses the same button label and the same dialog shell (date pickers vary by `scope_kind`). The word "Generate" is retired from user-facing copy; it lives on only as the internal function name in `nursery-scheduler.ts`.
