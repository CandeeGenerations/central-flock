# Fair Booth Schedule: dense per-hour signup grid as a third Schedule type

## Context

The church runs a 9-day Fair Booth ministry each year (county fair) and currently
tracks sign-ups on a hand-edited Google Doc: a dense per-hour grid of who's in
the booth from 2 PM to 10 PM across the run, plus a paired roster legend that
maps initials to names. Last year's PDF is in `data/fair-booth-schedule.pdf`.

The visible shape has features no existing Schedule type covers:

- **A timeline grid, not a Sunday-cell grid.** Each day is a column; rows are hours; cells are dense lists of initials annotated with role markers (dashes and stars) and partial-time annotations like `BA*(2-4)`.
- **Per-slot fixed structure with per-day variation.** Sat/Sun/Tue are split
  into two named slots (2-6 and 6-10); the other 6 days are one slot (5-10).
  Slot count is data, not a day-of-week rule.
- **Two person-level roles compounded with shift-local roles.** Stars on the
  initial denote what role the person can fill across the whole fair
  (1★ Worker → 5★ Fair Manager); dashes denote what role they're filling for
  that specific shift (1– Unit Leader, 2– Asst Unit Leader, 3– Worker).
- **Hispanic coverage colors the day header.** Green if a Hispanic person is
  present continuously across the day's hours, yellow/orange if partial, red
  if none. This is a load-bearing visual for the Fair Booth team specifically.
- **Headcount-driven row coloring with computed dotted lines** between hours
  where the headcount changes within a slot.
- **A roster legend on page 2.** Names + computed initials with collision
  resolution (`BC` for one, `BeC` for the other), counts per signup, with
  italic for "signed up to serve but not for a day yet" and bold for "signed
  up for < 3 slots."

Nursery is auto-generated by a fairness scorer; Special Music is a sparse 2-column
Sunday grid edited cell-by-cell. Neither covers a Gantt-style timeline with
shift roles, partial-time signups, headcount-driven coloring, or Hispanic
coverage. Three structural shapes were on the table:

- **Per-cell editing like Special Music.** Click each hour-cell, pick a person.
  Fails on dense days (Mon 11 at 7 PM has 7 entries); the cell is too small.
- **Manual schedule editor totally separate from the existing Schedule envelope.**
  Skip ADR 0006's `schedules` table. Reintroduces per-type plumbing — the exact
  thing 0006 was meant to retire.
- **Envelope + new body tables + day-editor sub-page.** Reuse the `schedules`
  row for envelope state; introduce `fair_booth_signups` (the body) and
  `fair_booth_roster_attrs` (sparse per-schedule attribute overrides for people
  on the live roster Group). The schedule detail page is read-only WYSIWYG;
  edits flow through a per-day sub-page route.

## Decision

**`schedule_type='fair_booth'` is added to the `schedules` envelope enum** as a
third type alongside `nursery` and `special_music`. Scope is `date_range`;
`scope_start` is always a Friday and `scope_end` is always 8 days later (the
following Saturday). The 9-day Fri→Sat shape is hardcoded; the slot pattern
(Sat/Sun/Tue 2-slot, rest 1-slot; hours 2–6, 6–10, 5–10) is hardcoded by
day-of-week. There is no `fair_booth_days` table — days are virtual, derived
from `scope_start`.

**Two new tables.** Drizzle / SQLite, in a new `server/db/schema-fair-booth.ts`:

```ts
export const fairBoothFairRoles = ['worker', 'asst_unit', 'unit_leader', 'asst_fair_mgr', 'fair_mgr'] as const
export const fairBoothShiftRoles = ['worker', 'asst_unit', 'unit_leader'] as const

export const fairBoothRosterAttrs = sqliteTable(
  'fair_booth_roster_attrs',
  {
    id: integer('id').primaryKey({autoIncrement: true}),
    scheduleId: integer('schedule_id')
      .notNull()
      .references(() => schedules.id, {onDelete: 'cascade'}),
    personId: integer('person_id')
      .notNull()
      .references(() => people.id, {onDelete: 'cascade'}),
    fairRole: text('fair_role', {enum: fairBoothFairRoles}).notNull().default('worker'),
    initialsOverride: text('initials_override'), // null = use computed
    createdAt: text('created_at')
      .default(sql`(datetime('now'))`)
      .notNull(),
    updatedAt: text('updated_at')
      .default(sql`(datetime('now'))`)
      .notNull(),
  },
  (t) => ({
    uniqSchedulePerson: uniqueIndex('fair_booth_roster_attrs_schedule_person_uniq').on(t.scheduleId, t.personId),
  }),
)

export const fairBoothSignups = sqliteTable('fair_booth_signups', {
  id: integer('id').primaryKey({autoIncrement: true}),
  scheduleId: integer('schedule_id')
    .notNull()
    .references(() => schedules.id, {onDelete: 'cascade'}),
  personId: integer('person_id')
    .notNull()
    .references(() => people.id, {onDelete: 'cascade'}),
  dayDate: text('day_date').notNull(), // 'YYYY-MM-DD'
  startMinute: integer('start_minute').notNull(), // minutes since midnight; 30-min granularity
  endMinute: integer('end_minute').notNull(),
  shiftRole: text('shift_role', {enum: fairBoothShiftRoles}).notNull().default('worker'),
  sortOrder: integer('sort_order').notNull().default(0),
  displayRowOverride: integer('display_row_override'), // null = auto; otherwise pinned to this row
  createdAt: text('created_at')
    .default(sql`(datetime('now'))`)
    .notNull(),
  updatedAt: text('updated_at')
    .default(sql`(datetime('now'))`)
    .notNull(),
})
```

**One new column on `people`.** `is_hispanic boolean not null default false`,
edited from the Person detail page. Used by the day-header coverage
calculation; intentionally person-level (it's the same value across years).

**Live roster, sparse attrs.** The roster pool is the live deduplicated union of
the Groups configured in `schedules.fairBooth.rosterGroupIds`. A person on
that pool with no `fair_booth_roster_attrs` row uses defaults
(`fair_role='worker'`, no initials override). A row is inserted lazily when
the user opens the roster modal and changes anything. Removing a person from
the Group makes them disappear from the page-2 roster list; existing signups
on the grid are preserved and surface as a soft warning ("X is no longer on
the roster Group but has 4 shifts").

**Signups FK the person directly,** not the roster attrs row. A signup is valid
for any person in `people`. The page-2 list is computed by querying the
roster Group's members and left-joining `fair_booth_roster_attrs` for this
schedule.

**At most 1 Unit Leader and at most 1 Asst Unit Leader per (schedule, day, slot).**
Enforced in the day editor and on save. Workers are unbounded and reorderable
within the tier via `sort_order` ↑↓.

**Initials.** Computed at render time from `firstName[0] + lastName[0]`. When two
people on the same schedule's roster would share the same 2-letter base, the
alphabetically-first person (by `lastName`, then `firstName`) keeps the base
form; later names extend by walking their first-name characters from position
2 and inserting a lowercased disambiguating slice between F and L (Becky
Candee → `BeC` when Brandon Cobb on the same roster keeps `BC`).
`initials_override` on `fair_booth_roster_attrs` always wins; overridden
people are excluded from the collision pool before the algorithm runs.

**No draft/final lifecycle for fair_booth.** Per ADR 0006, `schedules.status`
has values `draft` and `final` with exports gated behind `final`. Fair Booth
deviates: the schedule is always editable and exports are always available.
Rows stay at `status='draft'` permanently; the Fair Booth UI hides the status
badge and the Finalize/Reopen buttons from `<ScheduleActionsToolbar>`. The
column is not removed (other types still use it).

**Per-type settings**, all under existing ADR 0006 convention:

- `schedules.fairBooth.titlePrefix` (default `"Fair Booth Schedule"`)
- `schedules.fairBooth.rosterGroupIds` (multi-select of `groups`)
- `schedules.fairBooth.minSignupsForBold` (default `3`)
- `schedules.fairBooth.gridPageFooterBlocks` (footer-blocks JSON array;
  seeded with Proverbs 11:30 KJV as a bold `quote` and the original page-1
  instructions reworded for the footer)
- `schedules.fairBooth.rosterPageFooterBlocks` (footer-blocks JSON array;
  seeded with the original page-2 instructions reworded for the footer)

Logo uses the existing global `settings.schedulesLogoPath`.

**Routes:**

- `/schedules/fair-booth` — list page (filtered to `schedule_type='fair_booth'`)
- `/schedules/fair-booth/:scheduleId` — schedule detail. Side-by-side
  grid + roster on desktop; stacked on mobile. Grid cells clickable → day
  editor sub-page. Roster rows clickable → roster row modal.
- `/schedules/fair-booth/:scheduleId/day/:date` — day editor sub-page with
  prev/next-day chevrons, a live day-column preview, and a signups table.

**Export shape.** Two-page PDF (grid then roster), each page captured via the
shared `<SchedulePreviewFrame>`. JPG export captures page 1. **"Export blank
PDF"** is an additional affordance: renders page 1 with empty hour cells,
uncolored headers, no counts, and page 2 with the live roster as the legend
(since people sign initials, not names, on the printed sheet). Send-as-image
reuses `<SendScheduleDialog>`.

**Sidebar.** "Fair Booth" item joins the Schedules group between Special Music
and Settings.

## Why

- **Hard to reverse.** Two new tables and a new column on `people` enter the
  schema; the schedule envelope enum widens; the per-type settings convention
  picks up five more keys. Rolling back means a reverse migration on
  `fair_booth_*` plus dropping `people.is_hispanic`. A future contributor
  finding `people.is_hispanic` will (reasonably) wonder why a single ministry
  earned a `people` column when other ministry-specific facts didn't —
  recorded here as: it's coverage-bearing for the Fair Booth schedule header,
  it's identity-stable across years, and it's a single boolean. Worth the
  weight.

- **Surprising without context.**
  - **No `fair_booth_days` table.** Days are computed from `scope_start` via a
    hardcoded day-of-week → slot-pattern map. A reader seeing
    `fair_booth_signups.day_date` as a free `YYYY-MM-DD` will wonder why
    there's no parent row to FK; the answer is "the day shape is identical
    every year and the user has committed to that."
  - **Roster is live against a Group, but attrs are sparse-per-schedule.**
    Unlike RSVP lists (snapshot from Group) or Special Music (pure-live, no
    per-person attrs). The fair role and initials override are per-year
    facts (Pastor Candee may step down as Fair Manager; collisions may shift
    when the roster changes), so they live on a per-schedule join table; but
    the pool itself is live so mid-season adds/removes propagate.
  - **No draft/final.** ADR 0006 established the lifecycle as universal across
    Schedule types; Fair Booth opts out. The reasoning is operational: the
    sheet is reprinted weekly during the fair and circulated in churches to
    refill empty slots; gating exports behind "final" means flipping it back
    to draft every time anyone signs up. A future contributor adding another
    Schedule type should default to draft/final and only opt out with
    specific justification.
  - **`people.is_hispanic` is a column on a generic table for one feature.**
    Coverage drives the header color; it's not used elsewhere yet but plausibly
    could be (e.g., a "Hispanic ministry" Group seed, or other coverage-style
    schedules). Centralizing on `people` matches the location of other broadly-
    applicable person facts (`birthday`, `phone`); keeping it off
    `fair_booth_roster_attrs` avoids re-asking the same question every year.
  - **`fair_booth_signups.start_minute` / `end_minute` as integers, not
    HH:MM strings.** Lets headcount and overlap math work directly in SQL
    (`WHERE start_minute < :slot_end AND end_minute > :slot_start`). The trade
    is that the UI translates to/from minute-of-day at the form layer.

- **Real trade-off — alternatives rejected.**
  - **Per-cell editing à la Special Music.** Each hour-row × day-column cell
    is its own editor. Fails on density — a single cell on Mon 11 holds 7
    entries; the click target collapses to nothing. Also forces the user to
    enter the same person 5 times for a 5-hour shift. The day-editor sub-page
    keeps the signup as a single row with an explicit time range.
  - **A separate top-level Fair Booth feature outside the Schedules envelope.**
    Skip ADR 0006's `schedules` table; build a fresh `fair_booth_schedules` +
    `fair_booth_signups`. Lower migration risk (no `schedules.schedule_type`
    extension), no need to opt out of the draft/final lifecycle. Rejected
    because it duplicates list/detail/export/send plumbing again — the exact
    cost ADR 0006 was justified by avoiding. The "no draft/final" exception is
    cheaper than re-implementing the entire Schedule shell.
  - **Snapshot roster (like RSVP lists) instead of live + sparse attrs.**
    Copy Group members into a `fair_booth_roster` table at schedule creation;
    edit freely after. Rejected because the fair recruits people throughout
    the planning window; a snapshot would silently freeze out new
    Group adds, requiring a manual "re-sync from Group" affordance.
    Live-against-the-Group is what was asked for and what the singer-pool
    precedent already does.
  - **Per-schedule `min_signups_for_bold` column.** Per-schedule flexibility
    versus a single global setting. Rejected because the threshold reflects
    house style ("we expect 3 shifts from anyone serving") and shouldn't
    drift between years; global settings is the right shape.

## Consequences

- **`people.is_hispanic` is a new column on a high-traffic table.** Default
  `false`. Existing rows backfill to `false` and the user toggles per person on
  the detail page. Exports/import CSVs may need to handle the column to
  preserve round-trips.
- **The `schedules.status` column is now type-conditionally used.** Other types
  read it; `fair_booth` ignores it. Documented here so a contributor doesn't
  add a "status changed" feature assuming universal applicability without
  checking `schedule_type`. A future generalization could move
  `lifecycle: 'draft_final' | 'always_editable'` onto per-type settings if
  another type wants to opt out too — premature today.
- **The day-of-week → slot map is hardcoded.** Friday + Mon/Wed/Thu single-slot
  5–10; Sat/Sun/Tue two-slot 2–6 / 6–10. If a future fair changes shape
  (different start day, different hours, longer/shorter run), this becomes a
  code change. Acceptable in v1 because the local fair has used this pattern
  for years.
- **No FK from signups to `fair_booth_roster_attrs`.** A signup can exist for a
  person with no attrs row; render falls back to defaults. Removing the attrs
  row does not delete signups. Removing the person from the roster Group
  surfaces a soft warning in the schedule UI but does not delete the signup —
  the user resolves it manually (either re-add to Group, or delete the
  signup).
- **Initials are computed every render.** No `initials` column on
  `fair_booth_roster_attrs` (only `initials_override`). The collision pool
  is the live roster minus the override-holders, computed per request.
  Memoize at the route handler if perf shows up.
- **`fair_booth_signups.display_row_override` is row-position within a
  count-stable region, not a global row number.** The render algorithm groups
  hour rows by stable headcount (between dotted-line boundaries), then
  distributes signups across the region's rows; the override pins a signup
  to the Nth row of its region (0-indexed). When the region's bounds shift
  due to other signup edits, the override may end up out-of-range — render
  falls back to auto and the UI surfaces "row override no longer in range,
  reset to auto?"
- **Page 1 + page 2 are both clickable in the schedule detail view, but PDF
  capture is clean.** The clickability is invisible chrome (cursor: pointer,
  hover state at most); the captured DOM has no editor affordances. Mirrors
  the Calendar Print editor's tappable-cell pattern (ADR 0005). The PDF and
  the on-screen view are pixel-identical aside from cursor/hover.
- **The blank-PDF export is a separate render path**, not the live render with
  data zeroed. Headers are uncolored, count parens omitted, dotted lines
  absent. Page 2 is the live roster (it's the legend for the printed sheet
  and people use it to find their initials).
- **The 9-day, Fri→Sat constraint is validated on schedule creation.** The
  "starting Friday" date picker rejects non-Fridays. If the fair ever
  changes shape, this validation becomes an obstacle and the constraint
  needs to be revisited.
